# AI Rules — Fifteen Puzzles

Цей файл описує правила, яких повинен дотримуватись AI під час
аналізу, рев'ю, дебагу або генерації коду в цьому репозиторії.

AI повинен використовувати ці правила разом з файлами:

- `ARCHITECTURE.md`
- `AI_INDEX.md`

---

# 1. Загальні принципи проекту

Проект є мобільною грою "Fifteen Puzzles".

Основні принципи:

- проста та передбачувана архітектура
- чітке розділення відповідальності модулів
- детермінована логіка гри
- максимальна продуктивність UI
- мінімум прихованої магії
- мінімум великих рефакторингів без прямої потреби

AI не повинен пропонувати складні архітектурні рішення, якщо існує простіший і безпечніший варіант.

---

# 2. Архітектурні правила

Код проекту поділений на кілька систем.

## Game Logic

- модель дошки
- правила руху плиток
- логіка commit/apply
- відстеження empty cell

## Rendering

- Skia canvas
- плитки
- дошка
- графічні ефекти
- shader pipeline

## Gesture / Motion

- tap / pan
- axis lock
- drag-preview
- finish-анімація

## UI Layout

- React Native компоненти
- позиціонування
- екранна структура

## Infrastructure

- navigation
- fonts
- providers
- persistence / app shell

AI повинен поважати це розділення.

Логіка гри не повинна змішуватися з кодом рендерингу.

---

# 3. Канонічна модель ігрової дошки

AI повинен виходити з такої моделі:

- канонічний бойовий стан дошки зберігається у shared values
- `gridSV` є джерелом істини для позицій плиток
- `emptyRow` / `emptyCol` є канонічним положенням пустої клітинки
- React state не повинен бути джерелом істини для позиції плиток у hot path

AI не повинен повертати бойовий стан дошки назад у React state, якщо на це немає прямого й усвідомленого архітектурного запиту.

---

# 4. Ролі ключових вузлів

AI повинен трактувати відповідальність файлів так:

## `useGameBoardController.ts`

- канонічний контролер дошки
- створює shared values стану
- виконує `applyShift(...)`
- обробляє `onTapCell(...)`
- обробляє `onCommitShift(...)`
- тримає animation shared values
- повертає статичний список плиток для рендера

## `BoardGestureOverlay.tsx`

- gesture layer поверх canvas
- tap / pan arbitration
- axis lock
- drag-preview policy
- рішення commit / no-commit на `onEnd`

## `BoardTileNode.tsx`

- UI-side рендер конкретної плитки
- читає позицію з `gridSV`
- додає preview drag через `dragOffsetPx`
- додає finish-анімацію через animation shared values

## `GameBoardView.tsx`

- композиційний вузол
- збирає `Canvas`
- викликає controller
- монтує `BoardTileNode`
- монтує `BoardGestureOverlay`

AI не повинен знову переносити бойову логіку дошки в `GameBoardView.tsx`.

---

# 5. Правила рендерингу

Весь рендер ігрової графіки виконується через **Skia Canvas**.

React Native використовується тільки для:

- layout
- екранної структури
- контейнерів
- неігрових оболонок

AI не повинен пропонувати:

- React Native тіні для плиток
- `borderRadius` для ігрових графічних елементів замість Skia
- складні RN-анімації для плиток
- перенесення ігрового рендеру з Skia назад у RN Views

Графічні ефекти повинні реалізовуватись через Skia.

Складні векторні фігури з тінями (наприклад, внутрішні тіні) повинні реалізовуватись нативними можливостями Skia.

---

# 6. Правила продуктивності (Reanimated + Skia)

Продуктивність є критично важливою.

AI повинен уникати рішень, які можуть викликати:

- зайві React re-render
- складні layout recalculations
- heavy JavaScript loops під час анімацій
- логіку руху плиток у RN-render path

## Критичні правила інтеграції

1. **НЕ використовувати** `Animated.createAnimatedComponent()` для компонентів Skia (`Group`, `Rect` тощо).
2. Замість цього передавати результати `useDerivedValue` безпосередньо у властивість `transform` або інші нативні властивості компонентів Skia.
3. Drag плиток не повинен залежати від React re-render.
4. Жести повинні працювати через Reanimated/worklet pipeline.
5. Шейдери `.sksl` мають компілюватися **один раз** на рівні модуля через `Skia.RuntimeEffect.Make()`, а не створюватися під час рендеру.
6. Перехід між UI/runtime та RN runtime не повинен бути джерелом істини для руху плиток; це може бути лише побічний канал.

---

# 7. Правила drag / tap логіки

AI повинен виходити з поточної ігрової політики:

- стартова плитка визначає, яка група плиток буде рухатися
- `steps` означає **відстань стартової плитки до empty**, тобто розмір групи
- `steps` не означає "на скільки клітинок фізично протягнули палець"
- drag-preview візуально рухає групу лише на **один крок**
- commit виконується для **всієї групи**, якщо жест пройшов достатній прогрес
- preview, controller commit і finish-анімація повинні жити за однією математикою

AI не повинен пропонувати drag-preview на 2–3 клітинки, якщо controller і finish-анімація працюють як однокроковий зсув групи.

---

# 8. Правила організації коду

Кожен компонент повинен мати чітку відповідальність.

## UI компоненти

- відповідають за структуру
- не містять складної логіки гри

## Game logic

- знаходиться у відповідних модулях
- не повинна залежати від UI
- повинна бути детермінованою

AI повинен уникати створення:

- великих компонентів без потреби
- змішування логіки і UI
- дублювання джерел істини
- прихованої синхронізації між кількома станами, якщо можна мати один канонічний стан

---

# 9. Правила змін у коді

AI повинен пропонувати **мінімальні зміни**, необхідні для вирішення задачі.

AI не повинен:

- переписувати великі частини коду без потреби
- змінювати архітектуру без явного запиту
- додавати нові залежності без необхідності
- “покращувати все одразу”, якщо користувач просить точковий дебаг

Але:

Якщо користувач прямо просить **архітектурне рішення**, AI повинен чесно сказати, чи поточний фундамент придатний, а чи ні.

---

# 10. Правила дебагу

Під час дебагу AI повинен:

1. спочатку знайти канонічні файли для проблеми
2. перевірити, де саме знаходиться джерело істини
3. пояснити, чи проблема:
   - архітектурна
   - локальна логічна
   - математика жесту / анімації
   - UX / tuning
4. пропонувати правки малими кроками
5. після кожного кроку зберігати робочу базу

Для drag-дебагу порядок такий:

1. `BoardGestureOverlay.tsx`
2. `useGameBoardController.ts`
3. `BoardTileNode.tsx`

---

# 11. Правила документації

Документація проекту ведеться українською мовою.

AI повинен:

- відповідати українською
- використовувати англійську тільки для коду, назв API та технічних термінів, де це доречно
- не суперечити `AI_INDEX.md` та `ARCHITECTURE.md`

Якщо архітектура змінилася, AI повинен запропонувати оновити документацію, а не залишати її застарілою.

---

# 12. Взаємодія з AI

AI повинен:

- пояснювати запропоновані зміни
- посилатися на канонічні файли
- враховувати правила `ARCHITECTURE.md`
- враховувати карту файлів із `AI_INDEX.md`
- прямо попереджати, якщо запропоноване рішення порушує правила цього файлу

AI не повинен вдавати, що знає “канонічну” архітектуру, якщо користувач уже надіслав нові версії файлів.

У разі конфлікту між старою документацією та новими канонічними файлами, AI повинен спиратися на **поточні канонічні файли** і запропонувати оновити документацію.

---

# 13. Правила поточної робочої сесії

Коли починається новий чат по проекту, AI повинен орієнтуватися на:

- **Задача** — що саме робимо зараз
- **Файли в роботі** — які файли є канонічними для цього етапу
- **Проблема / Контекст** — на чому була зупинена робота
- **Поточна стабільна база** — яку версію не можна ламати без потреби

AI повинен поважати стабільну базу й не робити великий рефакторинг без окремого погодженого рішення.
